Map format sandbox — NE-Fan

Cuatro formatos para describir el mismo escenario (claro de taberna, 32×20 celdas a 2 m/celda). Edita cualquier textarea y pulsa "Re-render" para ver qué tal aguanta cada formato. El render común es deliberadamente plano — buscamos legibilidad del data source, no belleza visual.

A · ASCII monocapa

Una sola grid, un char por celda. Legend mapea char a terreno o entidad. El parser une clusters contiguos del mismo char en una entidad con bounding box.

Pros / contras
  • Ultrá compacto y "ves" el mapa en texto plano.
  • Fácil de modificar visualmente (mover una T una celda a la derecha).
  • Limitado a 1 char por celda — entidades con el mismo glyph se confunden si están adyacentes.
  • No hay metadatos por entidad (sólo lo que entre en el char).
—

B · ASCII doble capa

Dos grids del mismo tamaño: una de terreno, otra de entidades. Dos legends. Los chars de entidad se agrupan en clusters contiguos.

Pros / contras
  • Separa nítidamente el suelo de los objetos.
  • Permite poner cualquier entidad sobre cualquier terreno sin choque de chars.
  • Más verboso — dos matrices de 32×20.
  • Sigue sin metadatos ricos por entidad (legend tiene que cubrir el char).
—

C · JSON cells matrix

Una matriz 2D de objetos celda a celda con t (terreno) y opcional ent (id de entidad). Diccionario aparte para metadatos de entidades.

Pros / contras
  • Cada celda puede llevar cualquier atributo (humedad, transitabilidad, color…).
  • Sin ambigüedad — las entidades son objetos completos referidos por id.
  • Brutalmente verboso (640 entradas), ilegible al ojo desnudo.
  • Para un LLM: muchísimos tokens generados, alto riesgo de error y altísimo coste.
—

D · Híbrido (terreno ASCII + entidades como lista)

Terreno como grid ASCII legible, entidades como lista de objetos nombrados con cell, footprint y glyph. Coordenadas en celdas enteras.

Pros / contras
  • Suelo como mapa visual + entidades como objetos nombrados → ambos mundos.
  • Footprints > 1 celda, cada entidad con id único y metadatos arbitrarios.
  • Para el motor narrativo: trivial decir "mueve hooded_stranger a tavern_porch".
  • Dos representaciones que el LLM tiene que mantener coherentes (terreno + posiciones).
  • Si dos entidades caen en la misma celda, el render tiene que decidir cuál pinta encima.
